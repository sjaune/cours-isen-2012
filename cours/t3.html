Contexte et enjeux

Le marché du mobile

  • Explosion de l'utilisation de smartphones et des tablettes
  • 20 millions de smartphones pour 2 millions de tablettes
  • En 2011 : Plus de smartphones vendus que de mobiles classiques (11,9M contre 11,8M)
  • En 2014 le nombre d'utilisateurs d'Internet mobile devrait dépasser le nombre d'internautes standards

Profils des utilisateurs

De nouveaux usages

  • Utilisation en mobilité
    • 91% des utilisateurs gardent leur téléphone sur eux
  • Utilisation de fonctionnalités supplémentaires par rapport à un ordinateur standard :
    • Appareil photo / Caméra
    • Localisation (GPS)
    • Détection de l'orientation du terminal (portrait / paysage)
  • Un besoin rapide d'accéder à l'information

De nouvelles attentes

  • Un besoin rapide d'accéder à l'information
  • En France, seulement 14,3% des personnes sont prêtes à attendre plus de 5 secondes
  • Dans 20% des cas le site visualisé sur mobile est en erreur
  • Pire, dans 16% des cas la version mobile n'est même pas disponible
  • Le web mobile devrait fonctionner aussi bien que le web standard

De nouvelles contraintes ergonomiques

  • Nécessité de repenser les sites et applications pour ce format
  • Taille et résolution des écrans réduites
  • De nouvelles fonctionnalités sont disponibles :
    • Composition des numéros de téléphone depuis l'application
    • Géolocalisation de l'utilisateur
  • + toutes les règles du web actuel qui s'appliquent aussi

De nouvelles contraintes techniques

  • Diversité des terminaux à supporter :
    • Systèmes d'exploitation
    • Navigateurs Internet
    • Résolutions et tailles d'écran
  • Gestion de la connectivité (utilisation de la bande passante)
    • La bande passante n'est pas totalement gratuite sur les mobiles !
    • il faudra penser "léger" pendant le développement (optimisation des éléments, images, ...)

Application native, Web App mobile
C'est quoi la différence ?

L'application native

  • C'est une application développée spécifiquement pour des terminaux mobiles (smartphones, tablettes, …)
  • Elle est téléchargée (généralement depuis une boutique en ligne) sous forme d'un exécutable et installée sur le terminal
  • Elle est développée dans un langage spécifique à chaque plateforme
    • Java (Android)
    • Objective-C (iOS)
    • Visual C++ (Windows Mobile)

La Web App mobile ?

  • Web App = Web Application
    • pour les nuls en anglais = Application Web
  • Application accessible depuis le navigateur Internet du mobile
  • Programmée en HTML5 / CSS3 / Javascript
  • Hébergée sur un serveur web distant
  • Le langage de développement côté serveur est laissé au choix du développeur (PHP, Rails, Python, …)

Alors ? Web App ou Application native ?

Application native

Les Forces

  • Meilleure ergonomie et rapidité d'exécution
  • Exécutable compilé autonome
  • Installé directement sur le terminal
  • Utilisation des fonctionnalités hardware de la machine

Application native

Les Faiblesses

  • Langage de développement spécifique à chaque plateforme
  • Coût de développement plus élevé qu'une Web App

Web App mobile

Les Forces

  • Accessible depuis le navigateur
  • Cross platforms / cross devices
    • HTML / CSS et Javascript suffisent
    • Pas de nouveau langage à apprendre ... Ouf !
  • Mises à jours directement appliquées à tous les utilisateurs
  • Permet de tester son application avant de la porter en natif
  • Coût de développement raisonnable

Web App mobile

Les Faiblesses

  • Accès limité aux fonctionnalités hardware du terminal
    • Utilisation uniquement des fonctionnalités de l'API HTML5 et CSS3
    • Orientation, géolocalisation, vidéos, ...
  • Tests de compatibilités sur les OS et navigateurs difficile à assurer pour tous les modèles de terminaux

Et la réponse est...

Que le choix de la solution va directement dépendre du contexte projet !
(facile la réponse...)

Eléments à prendre en compte (entre autres) :

  • Contraintes de coûts et de délais
  • Compétences de développement disponibles
  • Contraintes techniques du projet (utilisation du hardware, application payante ou non, ...)

Le manifest

<html manifest="webapp.manifest">
  • Le fichier manifest gère la liste des éléments à mettre en cache sur le terminal
  • Utile pour développer un mode "offline" de son application (avec le cache d'application HTML5)
  • Cette déclaration n'est pas obligatoire

Le manifest

CACHE MANIFEST
# Commentaires
# v0.1
index.html
images/image1.png
css/styles.css
  • Extension de fichier .manifest
  • Type MIME text/manifest à déclarer au niveau du serveur web ou dans un .htaccess
  • Commence obligatoirement par CACHE MANIFEST
  • Les fichiers à mettre en cache sont déclarés ligne par ligne
  • Les commentaires commencent par un dièse (#)

Le manifest

Syntaxe complète

CACHE MANIFEST
# Commentaires
# v0.1

CACHE:
index.html
images/image1.png
css/styles.css

FALLBACK:
/ /offline.html

NETWORK:
*

Le manifest

Syntaxe complète

  • CACHE : section par défaut qui liste les fichiers à mettre en cache
  • NETWORK : liste les fichiers qui nécessitent obligatoirement une connexion internet
    • Le wildcard * = tous les autres fichiers
  • FALLBACK : liste les fichiers qui, au cas où ils ne soient pas accessibles en ligne, doivent renvoyer vers d’autres fichiers

Fonctionnalités CSS3

Généralités

  • Les terminaux mobiles utilisent (quasiment) tous les fonctionnalités de CSS3, alors pourquoi s'en priver !
  • Je vous invite donc à utiliser sans modération (avec parcimonie tout de même) :

    • et plein d'autres joyeusetés !

      • -webkit- : iOS Safari / Android Browser
      • -o- : Opéra

Le responsive design

responsive design = site web adaptable

Le responsive design

  • Ce n'est pas à proprement parler une fonctionnalité CSS3
  • C'est plutôt une technique de conception (HTML/CSS)
  • L'application et le contenu s'adaptent aux différentes tailles d'écrans et aux différents terminaux
    • Minimum de 480px et max de 640px pour les mobiles
    • 720px pour les tablettes
    • 960px max pour les versions desktop
  • Cette technique utilise massivement les media queries CSS

Les Media Queries

Une aubaine pour vos Web Apps !

  • Une réquête media consiste à cibler un type de média (ecran, print, ...) et une ou plusieurs conditions sur ses fonctionnalités (taille de l'écran, orientation, couleur, ...)
  • Les sites standards ne sont pas par défaut adaptés à la lecture sur mobile
  • On va donc faire appel à CSS3 et aux media queries pour y remedier pour charger des règles spécifiques
    • orientation : Orientation de l'écran (portrait, paysage)
    • min-width, max-width (idem avec height) : Résolution d'écran
    • device-aspect-ratio : Ratio d'affichage (4/3, 16/9)
    • device-pixel-ratio : Densité de pixels (pour les écrans Haute Qualité - le Retina Display d'Apple par exemple)
    • et bien d'autres à découvrir !

Les Media Queries

  • Caractéristiques des écrans des principaux terminaux mobiles :
    • 320x240, 240x320, 240x260
    • iPhone 3 / 3GS : 320x480
    • iPhone 4 / 4S : 640x960 (résolution x2 par rapport au 3GS)
  • et des tablettes :
    • iPad 1 / 2 : 768x1024 (132 ppp) - Ecran 9,7" (25cm)
    • iPad 3 : 1536x2048 (264 ppp) - Ecran 9,7" (25cm)

Les Media Queries

  • Pour cibler un terminal mobile particulier il faudra donc bien connaître ses caratéristiques :
    • iPhone : device-width: 320px
    • iPhone (Retina) : (device-width: 320px) and (-webkit-device-pixel-ratio: 2)
    • iPad (paysage) : (device-width: 768px) and (orientation: landscape)
    • ...
  • Attention toutefois, ces règles ne sont quand même pas très fiables :
    • Il se peut qu'un mobile (ou une tablette) dispose des mêmes caractéristiques qu'un autre
    • Il ne faut donc pas trop typer son code pour une marque spécifique (sauf si c'est volontaire)
    • Ou alors de préciser encore plus ces conditions en se tenant à jour des possibilités des derniers mobiles

Les media queries

Exemple de conditions dans un fichier CSS

/* iPhone (portrait et paysage) */

@media only screen
and (min-device-width : 320px)
and (max-device-width : 480px)
{
	body { background-image : url('bkg-iphone.jpg'); }
}

/* iPhone (landscape) */

@media only screen
and (max-width : 320px)
and (orientation : landscape)
{
	body { color: red; }
}
  • L'exemple ci-dessus affichera :
    • Un fond d'écran spécifique pour l'iPhone
    • Une couleur de texte rouge si en plus c'est au format paysage

Fonctionnalités HTML5

Les Touch Events

Les événements DOM liés au tactile

  • Nouveaux événements introduits par Apple dans iOS 2.0...
  • Puis implémentés sur Android...
  • Puis repris par le W3C pour en établir les spécifications!
  • Gobalement les équivalents aux "mouse events" sur les terminaux tactiles :
    • touchstart: un doigt est placé sur l'écran
    • touchmove : un doigt est déplacé sur l'écran
    • touchend : un doigt est enlevé sur l'écran
  • Certains frameworks en implémentent d'autres (tap, doubletap, drag, pinch, ...) en codant la logique

Les Touch Events

Exemple

<script type="text/javascript" charset="utf-8">
function init(){

     var myObject = document.getElementById("myObject");
     myObject.addEventListener('touchstart', function(e) {
	alert('Un doigt est posé !');
     }, false);

     myObject.addEventListener('touchend', function(e) {
	alert('Ah ben non il est levé...');
     }, false);
}
</script>

Les Gesture Events

La même chose mais avec deux doigts !

  • gesturestart: au moins deux doigts touchent l'écran
  • gesturechange : au moins deux doigts bougent sur l'écran
  • gestureend : au moins deux doigts quittent l'écran

Les Gesture Events

Exemple

<script type="text/javascript" charset="utf-8">
function init() {

	var myObject = document.getElementById("myObject");

	myObject.addEventListener('gesturestart', function(e) {
		alert("Zoom de départ: " + e.scale);
		alert("Rotation de départ: " + e.rotation);
	}, false);

	myObject.addEventListener('gesturechange', function(e) {
		alert("Zoom changé pour: " + e.scale);
		alert("Rotation changée pour: " + e.rotation);
	}, false);

	myObject.addEventListener('gestureend', function(e) {
		alert("Zoom de fin: " + e.scale);
		alert("Rotation de fin: " + e.rotation);
	}, false);

}
</script>

Les bonnes pratiques

Graphiques et ergonomiques

  • Baser son interface sur les terminaux avec la plus haute résolution disponible
  • Si possible utiliser des images vectorielles (SVG)
  • Storyboarder ses applications
    • Positionnement des éléments nécessaires
    • Comportement en mode portrait et paysage

Les bonnes pratiques

Techniques

  • Penser en terme de performances, codez léger !
  • Limiter le nombre et le poids des fichiers chargés
  • Utiliser des techniques I/O non bloquantes (Ajax, Workers, ...)
  • Utiliser les possiblité de cache du HTML5 (manifest, application cache, ...)
  • Suivre les bonnes pratiques générales de performances des sites Internet (compression des fichiers, nombre de requêtes, ...)

Conclusion

Le web mobile, avenir du web ?

  • Le marché du mobile et des applications web est florissant
  • Les versions adaptées pour mobile des sites Internet vont se démocratiser jusqu'à devenir un pré-requis au développement
  • De plus en plus de personnes seront connectées en dehors de leur ordinateur fixe (développement des mobiles 4G)
  • Les terminaux mobiles seront de plus en plus puissants et disposeront de plus en plus de fonctionnalités
  • Le web mobile viendra compléter l'utilisation fixe que l'on connaît (synchronisation des données, ...)
